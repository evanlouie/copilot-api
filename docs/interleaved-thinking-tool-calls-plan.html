Interleaved thinking & interleaved tool calls

A phased plan to surface streamed reasoning, preserve interleaved thinking across tool calls, and emit de-facto-standard reasoning / reasoning_content / reasoning_details fields on the OpenAI-compatible surfaces — grounded in a live probe of the GitHub Copilot SDK event stream.

Summary

The Copilot SDK already streams model reasoning as incremental deltas and exposes encrypted "opaque" thinking on every turn, but copilot-api collapses it into a single hidden TurnResult.Reasoning string that is never emitted, streamed, or replayed. This plan threads that reasoning through to the wire using the fields real clients already understand (Open WebUI, Jan, OpenCode, Pi), preserves interleaved thinking across the standard multi-request tool loop, and lifts the artificial parallel_tool_calls=true rejection.

Recommended strategy: change the internal event model in internal/copilotgw/turn_runner.go first (it is the choke point that flattens SDK events), then extend internal/openai wire types and the two HTTP encoders. Chat Completions is the primary target; Responses gets the higher-fidelity ordered-items mapping. Reasoning ordering is preserved by SSE chunk order — exactly how OpenRouter, Ollama, vLLM, and DeepSeek do it.

Reuse

The ordered SDK event callback, toolproxy batch/continuation model (already multi-call), session hardening, reasoning-effort resolution, and SSE writers.

Add

reasoning_delta stream events, reasoning / reasoning_content / reasoning_details wire fields, opaque-thinking capture, and an emission-policy knob.

Defer

Portable stateless store:false encrypted-reasoning round-trip (SDK strips opaque reasoning on resume); document as a known limitation.

Spike evidence

A live probe instrumented the real turn_runner event loop (COPILOT_SPIKE_TRACE=1) and drove two turns through the gateway against claude-sonnet-4.6 with reasoning_effort=high and a get_weather tool. The raw copilot.SessionEvent ordering was captured and the instrumentation reverted afterward.

FindingEvidenceConsequence
Reasoning streams incrementally AssistantReasoningDeltaData{DeltaContent, ReasoningID} — 36 then 38 deltas across two turns. Live delta.reasoning is achievable; the loop just has no case for it today.
Interleaved thinking across tools is real Turn 2 emits a new reasoning block (distinct ReasoningID) before its message, after the tool result. Preserved automatically by the standard Chat multi-request loop — no special transport needed.
Opaque/encrypted thinking is exposed AssistantMessageData.ReasoningOpaque != nil on every turn (Anthropic signature blob). reasoning_details with signed/encrypted continuity is feasible, not just plaintext.
Within a turn: reasoning, then text, then tool Reasoning deltas complete before AssistantMessageStart; tool request is in the final message. Emit reasoning chunks before content/tool chunks (Ollama's split pattern).
AssistantStreamingDeltaData is noise Carries only TotalResponseSizeBytes (a byte-count heartbeat). Ignore it explicitly; do not treat as content.

Wire contract

Ordering between reasoning, content, and tool-call deltas is conveyed entirely by SSE chunk order. Reasoning continuity round-trips by replaying the assistant message (including reasoning fields) before tool results — the same mechanism documented by OpenRouter, DeepSeek, and Ollama.

Implementation phases

Internal event model — capture & stream reasoning

Change the choke point so reasoning is no longer dropped. This is the prerequisite for every wire change.

  • In internal/copilotgw/turn_runner.go, add case *copilot.AssistantReasoningDeltaData → emit a new internal reasoning-delta event.
  • On AssistantMessageData, capture ReasoningText, ReasoningOpaque, and EncryptedContent into TurnResult (new fields).
  • Explicitly ignore AssistantStreamingDeltaData (heartbeat).
  • In internal/copilotgw/types.go, add StreamEvent{Kind:"reasoning_delta", Delta, ReasoningID} and reasoning/opaque fields on TurnResult.

OpenAI wire types — reasoning fields

Add the de-facto-standard fields and a structured detail type; accept them inbound tolerantly.

  • In internal/openai/types.go, add reasoning, reasoning_content, and reasoning_details []ReasoningDetail to the assistant message and the chunk delta.
  • Define ReasoningDetail{Type, Text, Signature, Data, Summary, ID, Format, Index} with omitempty semantics per type.
  • Parse inbound assistant-message reasoning without erroring (tolerate reasoning/reasoning_content/reasoning_details).

Chat Completions encoder — interleave & aggregate

Primary surface. Stream reasoning before content/tool deltas; attach aggregate reasoning to the final message.

  • In internal/httpapi/chat.go, handle reasoning_delta → emit delta.reasoning (+ delta.reasoning_content) chunks, ordered ahead of content/tool deltas (Ollama split pattern).
  • On the terminal message, populate reasoning, reasoning_content, and reasoning_details alongside content and tool_calls.
  • Wire the emission-policy knob: both (default) | reasoning | reasoning_content | off.

Validation — accept parallel tool calls

Remove the artificial rejection; keep structural validation intact.

  • In internal/openai/validation.go, drop the parallel_tool_calls=true Chat rejection.
  • Confirm toolproxy multi-call batch + Broker.FindByCallIDs behavior is unchanged; add a regression test.

Responses surface — ordered reasoning items

Higher-fidelity mapping for /v1/responses.

  • In internal/httpapi/responses_stream.go, emit ordered reasoning output items and reasoning stream events alongside function_call items.
  • Carry reasoning_details-equivalent content into the reasoning item (content + encrypted_content when present).
  • Preserve item order and stable IDs from the SDK ReasoningID.

History hydration & continuation

Make the round-trip lossless without depending on the client.

  • In internal/hydration/hydration.go, tolerate inbound assistant reasoning; replay it only when reconstructing a cold/stateless session.
  • Prefer the live Copilot session for continuity; never 400 on client-supplied reasoning.
  • Document that portable store:false encrypted-reasoning round-trip is unsupported (SDK strips opaque reasoning on resume).

Tests, docs & config

Lock behavior and surface the new knobs.

  • Unit tests: reasoning-delta mapping, final-message aggregation, reasoning_details shape, parallel tool_calls acceptance, emission-policy knob.
  • Formalize the throwaway probe as a gated live test (COPILOT_API_LIVE_TESTS=1) asserting reasoning deltas precede content.
  • README compatibility matrix + new env var for the emission policy.

Testing plan

Deterministic (unit)

  • Fake SDK event sequence → assert ordered reasoning_delta then content then tool_calls.
  • Final message carries reasoning/reasoning_content/reasoning_details.
  • Inbound reasoning tolerated; cold-history replay path.
  • parallel_tool_calls=true accepted; multi-call batch preserved.

Live (gated)

  • Reasoning streams as deltas before content (claude-sonnet-4.x, reasoning_effort high).
  • Second turn produces a fresh reasoning block after a tool result.
  • ReasoningOpaque present and surfaced into reasoning_details.

Risks & gotchas

Duplicate reasoning display. Some clients render both reasoning and reasoning_content. Default to emitting both, but ship the narrowing knob and validate against Open WebUI / Jan / OpenCode / Pi.
Opaque reasoning is session-bound. The SDK strips ReasoningOpaque/EncryptedContent on resume. Portable stateless continuity would require our own encryption envelope — out of scope for v1; document the limitation.
Provider drift. vLLM moved reasoning_content → reasoning; keep an adapter seam rather than hard-coding one field.
Client-proxied tool execution. The SDK's own tool handler reports success=false for parked client tools (expected). Interleaving across tools happens across HTTP requests, not inside one stream.

Acceptance criteria

Must ship

  • Streaming Chat emits ordered reasoning deltas before content/tool deltas.
  • Non-streaming Chat attaches reasoning/reasoning_content/reasoning_details.
  • Interleaved thinking preserved across the tool-call loop.
  • parallel_tool_calls=true accepted on Chat.
  • Emission-policy knob with a sane max-compat default.
  • Deterministic tests + gated live test green; README updated.

May defer

  • Responses ordered reasoning items (Phase 5) if scheduled separately.
  • Portable store:false encrypted-reasoning round-trip.
  • Per-format reasoning_details normalization beyond Anthropic.
Definition of done: a Chat-Completions client receives streamed, ordered reasoning interleaved with tool calls, can replay it for continuity, and sees coherent interleaved thinking across a multi-step tool loop — documented in the README compatibility matrix and covered by deterministic and gated live tests.
