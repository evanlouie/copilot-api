Implementation plan · Responses API · Dynamic tool search

Install tool_search results through an explicit SDK turn boundary

A plan for Option B: when Codex returns loadable tool specs from tool_search_output.tools, merge them into the request-scoped Responses tool catalog and continue in a newly configured Copilot SDK turn, so deferred tools such as multi_agent_v1.spawn_agent become actually callable without making copilot-api a proxy-hosted tool runtime.

Summary

Codex can currently call tool_search and return loadable tool specs, but those returned specs are only validated and recorded as output text. They are not installed into the Copilot SDK session's active tool catalog. The result is the exact failure mode observed with sub-agents: Codex discovers multi_agent_v1.spawn_agent, wait_agent, send_input, resume_agent, and close_agent, but the model cannot call them in the subsequent step because the SDK turn still has the old tool set.

Strategy: make tool_search_output a catalog boundary. When live search output contains valid loadable tools, validate ownership and mergeability first, persist the client-supplied search output as continuation transcript state, cut over from the old parked SDK turn without exposing stale-catalog results, and start exactly one next model turn with a new RequestTools instance built from the merged catalog.

The proxy should expose these loaded tools to the model and return their calls to Codex CLI. Codex CLI remains the executor. copilot-api must not run sub-agents, MCP actions, shell commands, or any other client-owned tool logic.

Current gap

The current implementation has the necessary parsing and safety foundation, but not the dynamic catalog step:

  • tool_search_call is exposed as a client-owned tool.
  • tool_search_output is parsed and its tools array is validated as loadable function or namespace specs.
  • The returned specs are not merged into ResponseRequest.Tools for the next SDK turn.
  • The live parked SDK handler receives only a string output; the session's Tools and AvailableTools remain unchanged.
  • Completing that parked handler would let the old runner continue planning with the stale catalog unless the implementation explicitly cuts it over.
  • Warm sessions and stored response records do not yet carry an installed-catalog delta produced by tool search.
Observed effect: the model can talk about discovered tools from the search result, but when it tries to call multi_agent_v1.spawn_agent the tool is absent from the SDK-configured callable set.

Goals and non-goals

Goals

  • Install valid tool_search_output.tools for the model's next tool-planning step.
  • Support deferred namespace tools such as multi_agent_v1.spawn_agent.
  • Preserve canonical Responses names and namespaces in output items.
  • Persist enough catalog state for subsequent continuations, warm sessions, and restart fallback.
  • Keep the SDK in ModeEmpty with only request-scoped synthetic custom tools exposed.

Non-goals

  • Do not execute sub-agent tools inside copilot-api.
  • Do not enable Copilot SDK built-in MCP, shell, file, memory, skills, repository, or host-git tools.
  • Do not implement hosted OpenAI tools such as provider web search, file search, image generation, or code interpreter.
  • Do not mutate the active SDK tool list mid-turn unless the SDK gains a supported API for it.

Architecture

The dynamic path becomes search → validate active outputs → merge catalog → persist transcript/catalog → cut over old runner → reconfigure SDK → continue.

Turn N SDK catalog
  └─ tool_search
       ↓ model calls tool_search
Responses output
  └─ tool_search_call(call_search)
       ↓ Codex executes search and returns tools
Continuation input
  └─ tool_search_output(call_search, tools: [namespace multi_agent_v1 ...])
       ↓ proxy validates active outputs, status, ownership, allowlisted fields, and collisions
Merged catalog
  ├─ original function/custom/namespace tools
  ├─ tool_search
  └─ loaded namespace multi_agent_v1.*
       ↓ old runner is aborted/drained; new synthetic or proven-safe resumed SDK turn is configured with merged catalog
Turn N+1 model can call multi_agent_v1__spawn_agent
       ↓ proxy rehydrates
Responses output
  └─ function_call(namespace: "multi_agent_v1", name: "spawn_agent")
Important boundary: the new tools become callable only after a new SDK turn is configured with the merged catalog. The implementation must never claim that returned tools were callable before Tools and AvailableTools actually include them.

Implementation phases

Phase 1

Extract one canonical tool-catalog module

Create a single source of truth for normalization, merge, aliasing, collision checks, semantic keys, and persistence DTO conversion.

  • Add a ToolCatalog type around canonical Responses tool specs, not raw SDK aliases.
  • Expose operations: FromRequest, FromStored, MergeLoaded, Flatten, Key, StoredDTO, and WithoutRaw.
  • Move duplicate SDK-safe alias and collision logic out of both openai and toolproxy into this module or a shared helper.
  • Keep provenance such as source search call_id outside the semantic catalog key unless intentionally comparing audit history.
  • Ensure catalog keys ignore raw JSON byte ordering/whitespace but preserve schema, format, strictness, defer-loading, namespace, execution, and descriptions.
  • Represent “catalog missing from migrated record” distinctly from “known empty catalog.”

Phase 2

Define safe persisted DTOs and data limits

Do not persist Go-internal structs with transient fields as the durable schema.

  • Bump sessionstore.ResponseRecordVersion.
  • Add explicit JSON DTOs such as StoredToolCatalog, StoredToolSpec, StoredLoadedToolEvent, and StoredToolOutput.
  • Persist canonical Responses identities and meaningful schema/format fields; do not persist SDK aliases as source of truth.
  • Persist raw returned tool_search_output.tools only where needed for transcript fidelity; default catalog storage should be WithoutRaw.
  • Cap loaded tool count, per-tool schema bytes, total catalog bytes, nesting depth, and description length before storage or SDK exposure.
  • Log loaded-tool catalogs as counts and canonical names only; never log full schemas/descriptions except under existing explicit content logging and truncation policy.
  • Document that dynamic catalogs may contain sensitive tool descriptions/schemas and are retained locally until purge, including for store:false continuation/debug state.
Phase 3

Parse returned tools into typed, allowlisted continuation outputs

Stop treating tool_search_output.tools as only raw JSON after validation.

  • Extend the internal continuation output carrier with LoadedTools []NormalizedTool, LoadedCatalogDelta, or an equivalent typed field.
  • Allowlist fields for returned function and namespace specs; reject or strip unknown provider-hosted fields such as server_url, server_label, auth/provider configuration, hosted execution hints, and any hosted web/file/code/image tool shape disguised under a client-owned type.
  • Reject returned custom, nested tool_search, hosted tools, provider-hosted MCP declarations, nested namespaces, malformed schemas, sentinel collisions, duplicate identities, and SDK alias collisions.
  • Install loaded tools only when the matched pending call is a live tool_search call with execution: "client" and status absent, success, or completed.
  • If status is failed, incomplete, cancelled, or unknown, record the search output but do not install accompanying tools; reject if the client tries to combine a failed status with loaded tools unless a harmless no-install mode is explicitly chosen.
  • Track the source call_id, response id, raw tools payload, normalized loaded tools, and merge result for audit/replay.
Phase 4

Resolve the effective catalog for every continuation

Continuations need a deterministic catalog source before any SDK turn starts.

  • On initial requests, use the normalized request catalog and persist it as the installed catalog.
  • On previous_response_id, start from previousRecord.InstalledToolCatalog when present.
  • For migrated records without catalog state, use the current request tools only when explicitly supplied; otherwise mark loaded-tool replay unavailable and fail clearly if a dynamic loaded tool would be required.
  • Merge active tool_search_output.LoadedTools into the previous installed catalog before completing or cancelling the live batch.
  • Define repeated-search behavior: identical already-installed tools are idempotent, exact semantic duplicates are accepted as no-ops, conflicting definitions reject, and multiple search outputs in one batch merge in deterministic call_id order.
  • Handle empty tools arrays as successful searches with no catalog change.
  • Honor effective tool_choice: "none" on continuation by withholding all installed tools and exposing only the sentinel, even if a loaded catalog exists.
Phase 5

Create a shared continuation planner

A single gateway-level planner should drive non-streaming HTTP, streaming HTTP, WebSocket, and warm-session paths.

  • Take parsed outputs, request metadata, optional warm session, previous response record, and live batch lookup as input.
  • Return a decision: normal parked-handler continuation, dynamic tool-search cutover, cold synthetic fallback, warm response creation, or clear unsupported request error.
  • Derive previous_response_id from the live batch when omitted and assign the continuation response id exactly once.
  • Select active outputs from the current live batch while preserving stale stateless-history outputs for transcript reconstruction only.
  • Validate model, previous-response ownership, endpoint kind, call kind, optional custom output name, search status/execution, and exact output count before mutating any live batch.
  • Ensure non-streaming and streaming code paths save equivalent final response records and pending-batch ownership.
Phase 6

Implement explicit old-runner cutover

The dynamic branch must not let the old stale-catalog runner produce a client-visible post-search turn.

  • Validate active outputs and merge the loaded catalog before completing, cancelling, or removing the batch.
  • Persist the client-supplied tool_search_output in a structured continuation transcript/log, not as an API-visible assistant output item.
  • Default implementation: cancel/abort/disconnect the old runner, remove the broker batch, forget the pending runner, and drain/discard any late old-runner results.
  • Do not call the normal CompleteToolOutputsWithSetup path for loaded-tool search outputs unless a gated integration test proves the SDK can record the tool result and then be safely resumed with replaced tools before stale-catalog planning occurs.
  • Assert no duplicate final response, no remembered old runner, and no live batch leak after cutover.
Default to synthetic cutover: until real SDK behavior proves that resume can replace tools safely after a delivered tool result, build the post-search continuation from stored transcript state in a fresh/synthetic SDK turn configured with the merged catalog.
Phase 7

Build replay prompts for synthetic and resumed modes

Continue. is only sufficient when the same SDK session already contains the full prior conversation and accepted tool result.

  • For synthetic cutover, include prior user/assistant transcript, the assistant tool_search_call, all submitted tool outputs in the active batch, returned tool specs or a concise availability note, and any mixed user suffix after outputs.
  • For proven-safe same-session resume, use a minimal continuation prompt only after the SDK has recorded the tool result and the new catalog is installed.
  • Keep API-visible response.output as assistant outputs only; store client continuation inputs/tool outputs in separate continuation transcript fields.
  • Include canonical names in transcript notes and avoid SDK aliases in prompts unless needed for debugging under explicit content logging.
  • Preserve namespaced function calls, custom inputs, tool-search arguments, tool-search status/execution, and raw returned tool summaries in cold fallback.
Phase 8

Implement transport support incrementally

Sequence transports so each path has deterministic proof before broad rollout.

  • First: non-streaming POST /v1/responses with fake SDK/provider capture proving merged Tools and AvailableTools.
  • Second: HTTP SSE streaming with identical planner decision and stable lifecycle events.
  • Third: WebSocket response.create using the same gateway-level planner and sequence-number semantics.
  • Fourth: generate:false warm sessions. Define whether generate:false with tool-output continuations warms the post-search turn through the merge planner or fails clearly until supported.
  • If any transport is not completed in the first implementation slice, add a temporary explicit feature gate/error rather than silently falling back to stale catalogs.
Phase 9

Harden safety boundaries

Installing dynamically discovered tools must not weaken the proxy's no-execution invariant.

  • Keep ModeEmpty.
  • Keep SDK runtime defaults with MCP servers, skills, shell/file/repository/memory operations, host-git operations, sub-agent streaming, and session store disabled.
  • Permissions approve only exact SDK names generated from the merged request-scoped catalog.
  • Reject any SDK tool request not present in the current request-scoped catalog.
  • Do not install hosted or proxy-executed tools returned by tool_search_output.tools.
  • Tests should prove copilot-api never spawns sub-agent processes, shell commands, MCP servers, file tools, or SDK built-ins as a result of dynamic installation.
Phase 10

Codex sub-agent proof and documentation rollout

Prove the observed failure mode is fixed without depending on nondeterministic model output.

  • Use a fake SDK/provider that emits a loaded-tool call only when the merged catalog actually contains multi_agent_v1__spawn_agent.
  • Assert the proxy rehydrates that call as namespace: "multi_agent_v1", name: "spawn_agent".
  • Keep Codex CLI as a manual/non-gating smoke test: discover multi_agent_v1, call a harmless sub-agent operation, wait, and close.
  • Update README once automatic post-search installation lands; remove the current limitation that returned tools require a later request with an installed catalog.

    Version              int

}

HTTP and WebSocket flows

Flow Required behavior
Initial HTTP response Normalize request tools, store installed catalog and semantic key, expose tool_search and any non-deferred tools.
Live HTTP continuation with search output Planner validates active outputs, resolves effective catalog, merges loaded tools, persists tool-output transcript and catalog, cuts over old runner, starts one synthetic/proven-safe resumed SDK turn with merged catalog, and returns the next model response.
Streaming continuation Use the same planner decision as non-streaming; emit lifecycle events for the post-search turn only, with stable response id, sequence numbers, output indexes, and pending-batch ownership.
WebSocket continuation Route response.create through the same gateway-level planner; preserve sequence numbers, one-active-response rules, and top-level error behavior.
generate:false warm session Either support tool-output continuations through the same merge planner and warm the post-search turn, or fail clearly until supported. Warm reuse compares semantic catalog keys.
Cold fallback / restart / TTL Resolve catalog from the previous response record; reconstruct a transcript with prior assistant calls, client tool outputs, search results, loaded-tool availability, and mixed user suffix before configuring the synthetic turn.

Test plan

Unit tests

  • Catalog merge with functions and namespaces, including idempotent duplicate loads.
  • Collision rejection against existing tools, loaded tools, aliases, and sentinel.
  • Canonical catalog keys ignore raw JSON order but preserve meaningful fields; provenance is excluded from semantic equality.
  • Returned hosted/custom/tool_search/nested namespace specs reject.
  • Returned spec field allowlist rejects provider-hosted or auth/server fields disguised in function/namespace specs.
  • Status handling: completed installs; failed/incomplete does not install.
  • Permission handler only approves merged request-scoped SDK aliases, and tool_choice:none exposes only sentinel.

Gateway tests

  • Live tool_search_output with loaded namespace starts exactly one post-search turn with merged AvailableTools.
  • The old runner is cancelled/drained, broker batch removed, and no stale-catalog result is emitted.
  • The next model tool request for multi_agent_v1.spawn_agent rehydrates as namespaced function_call.
  • Wrong output kind/name/status/execution and wrong previous response id fail before state mutation.
  • Cold fallback reconstructs search output, loaded-tool availability, and effective catalog from stored record state.

Streaming, WebSocket, and warm tests

  • Streaming continuation emits lifecycle events for the post-search turn only, with stable response id, sequence numbers, and output indexes.
  • WebSocket response.create supports search-output merge and loaded-tool calls through the same planner.
  • generate:false warm-session behavior is either supported through merge planner or rejected clearly.
  • Warm session reuse/mismatch covers merged catalogs and semantic catalog keys.

Deterministic smoke and safety tests

  • Fake SDK/provider records exact merged SDK tool names and filters, and emits loaded-tool calls only when present.
  • Aliases do not appear in SSE/WS events, ResponseRecord.Output, fallback transcripts, or normal log summaries.
  • No SDK built-ins, sub-agent processes, shell/MCP/file tools, or host-git operations are enabled after dynamic install.
  • Manual Codex CLI smoke: discover multi_agent_v1, call a harmless sub-agent operation, wait, and close.

Risks and mitigations

Risk Mitigation
Old SDK turn continues with stale catalog Default to synthetic cutover: validate/merge/persist, abort or drain old runner, remove batch, then start one merged-catalog turn.
SDK resume does not actually replace tools Do not rely on resume replacement until live integration proves it. Use synthetic continuation by default; gate resume optimization separately.
Catalog state is unrecoverable after restart Persist versioned installed catalog, loaded-tool events, and structured tool outputs on response records.
Loaded tools accidentally become proxy-hosted Keep handlers parked, permissions scoped, SDK built-ins disabled, and returned spec allowlists strict.
Dynamic catalogs leak sensitive schemas/descriptions Persist minimized canonical specs, cap sizes, truncate logs, and document local retention/purge behavior.
Catalog state drifts between HTTP, WebSocket, and warm sessions Use one shared planner and one shared catalog module for all transports.
Alias behavior changes model planning or leaks outward Preserve canonical names in descriptions and rehydrate outward names. Add tests that aliases do not appear in API-visible outputs or normal logs.

Acceptance criteria

  1. A live tool_search_output containing a multi_agent_v1 namespace causes the next SDK turn to include synthetic custom tools for its child functions.
  2. The old stale-catalog runner is cancelled/drained/discarded and cannot emit a duplicate or stale final response.
  3. The model can call a loaded tool such as multi_agent_v1.spawn_agent after the boundary.
  4. The proxy emits function_call with namespace: "multi_agent_v1" and name: "spawn_agent", not the SDK alias.
  5. Codex CLI executes the loaded tool and continues with function_call_output; the proxy never executes it.
  6. Returned loadable tools are persisted in canonical catalog state and survive subsequent continuations and restart fallback.
  7. HTTP non-streaming, HTTP streaming, WebSocket, and warm-session paths either use the same catalog merge planner or explicitly fail behind a documented temporary gate.
  8. Returned hosted/proxy-executed tools, provider-hosted fields, nested namespaces, returned custom tools, and returned nested tool-search tools are rejected.
  9. tool_search_output.status failed/incomplete does not install tools.
  10. tool_choice: "none" with an installed dynamic catalog exposes no callable tools in that turn.
  11. No Copilot SDK built-in tools, proxy-hosted MCP servers, shell/file/repository tools, host-git operations, or sub-agent processes become available after dynamic tool installation.
  12. Tests prove tools are not callable before the boundary and are callable after the merged catalog is installed.
  13. README is updated to remove the old limitation that returned tool_search_output.tools require a later request with an installed catalog.
