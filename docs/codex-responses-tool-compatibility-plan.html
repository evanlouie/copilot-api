Implementation plan · Responses API · Codex CLI

Surface Codex’s full Responses tool set through copilot-api

A staged plan to support the non-function Responses tools that Codex CLI expects—freeform custom tools such as apply_patch, namespaced MCP tools, and tool_search—while keeping the proxy’s core safety invariant: client-owned tools are never executed by the proxy.

Summary

The proxy currently accepts Codex-shaped Responses requests in permissive mode, but it only forwards type: "function" tools into the GitHub Copilot SDK. That filters out the tools Codex models rely on for a complete coding-agent experience: apply_patch as a freeform custom tool, MCP tools grouped under Responses namespaces, and the tool_search discovery tool.

Implementation strategy: introduce a normalized Responses tool catalog, flatten every supported client-owned tool into SDK-compatible custom tools, capture SDK tool invocations with enough metadata to rehydrate the original Responses output item shape, and extend continuation parsing for the corresponding output items.

This does not require enabling Copilot SDK built-in shell/file/MCP tools. The SDK remains a model transport; Codex CLI remains the tool executor.

Current state

The relevant implementation paths are narrow and intentionally conservative:

  • internal/openai/types.go models a tool primarily as Type plus nested Function.
  • internal/openai/validation.go lets non-function Responses tools pass in permissive mode but skips them.
  • openai.SupportedTools returns only tool.Type == "function".
  • internal/httpapi/responses.go passes only SupportedTools(req.Tools) into the gateway.
  • internal/toolproxy/toolproxy.go registers only those function tools as SDK custom tools.
  • README.md documents the current behavior: non-function Responses tools emitted by Codex CLI are ignored in permissive mode and rejected in strict mode.
Observed effect: a Codex CLI request may contain all tools, but the model only sees the subset that survived SupportedTools. The missing entries are therefore missing from model behavior, not from Codex’s local tool runtime.

Architecture

The core pattern is normalize → flatten → invoke → rehydrate → continue.

1. Normalize

Parse the incoming Responses tools array into a typed catalog that preserves raw JSON, public names, namespace metadata, output kind, and validation state.

2. Flatten

Convert every supported client-owned tool into a Copilot SDK custom tool with a safe internal SDK name and a mapping back to its Responses identity.

3. Invoke

Let the model call SDK custom tools. The handler still parks the turn and waits for Codex CLI to execute the real tool.

4. Rehydrate

Transform captured SDK invocations back into the correct Responses output item type: function_call, custom_tool_call, or tool_search_call.

Implementation phases

Phase 0

Capture the exact Codex request and model-discovery shape

Before changing behavior, add local reproduction fixtures and request-shape logging so the implementation is grounded in real Codex CLI payloads.

  • Add a redacted debug log under COPILOT_LOG_CONTENT that counts tool types and prints names/namespaces, not full schemas by default.
  • Add a fixture based on the user-observed list: a regular function tool, dotted function names such as multi_tool_use.parallel, custom apply_patch, a namespace with child tools, and top-level tool_search.
  • Fixture exact wire shapes: custom with name, description, and format {type, syntax, definition}; namespace with name, description, and child tools; and tool_search with execution, description, and JSON-schema parameters.
  • Confirm whether missing tools are present in incoming /v1/responses requests or suppressed earlier by Codex model metadata.
  • Capture which model endpoint Codex CLI calls (/v1/models, /codex/models?client_version=..., or another configured path) before adding any metadata compatibility mode.

Phase 1

Introduce a Responses tool catalog

Replace the function-only helper path with a richer representation for Responses tools while keeping Chat Completions function-only.

  • Add a new type such as openai.ResponsesTool or extend openai.Tool with raw fields for function, custom, namespace, and tool_search.
  • Create NormalizeResponsesTools(req.Tools, strict) that returns supported client-owned tools plus explicit ignored/unsupported diagnostics.
  • Keep ValidateTools for Chat unchanged; update ValidateResponsesTools to validate supported Responses-only types in both strict and permissive modes.
  • Preserve strict, defer_loading, format, execution, namespace child metadata, and raw JSON so tool_search_output.tools can be round-tripped accurately.
  • Define and test the strict/permissive validation matrix before changing runtime behavior.
Phase 2

Flatten supported tools into SDK custom tools

Extend internal/toolproxy so each normalized Responses tool becomes one SDK custom tool, or multiple SDK tools for namespace children.

  • Define a ClientTool metadata struct with SDKName, Kind, ResponseName, Namespace, Description, Parameters, Strict, DeferLoading, and original raw spec.
  • Use deterministic SDK-safe aliases for every tool kind when the public Responses identity is not SDK-safe. This includes dotted function names such as multi_tool_use.parallel, not only namespace children.
  • For SDK-safe direct functions, keep the public name only when it does not collide with another encoded alias or reserved sentinel.
  • For custom, synthesize an SDK JSON schema with a required string input field and a description explaining the downstream freeform shape.
  • For namespace, flatten each child into a deterministic safe SDK name, for example mcp__grep_app__searchGitHub, and store the Responses namespace separately.
  • For tool_search, expose an internal SDK alias for the canonical top-level tool_search tool using the provided query/limit schema. The alias must never leak into Responses output items.
  • Detect and reject collisions among direct safe names, encoded aliases, returned loadable tools, and reserved names such as __copilot_api_no_tools__ before the session starts.
Important: the SDK transport currently exposes JSON-schema custom tools, not native OpenAI freeform grammar tools. The adapter should preserve Codex’s output shape, but the model-facing prompt/schema may need a small compatibility shim for freeform custom tools.
Phase 3

Introduce generic captured-call and continuation-output plumbing

The current pipeline carries only []openai.ChatToolCall and map[string]string function outputs. Extended Responses tools need typed call and output carriers across httpapi, copilotgw, and toolproxy.

  • Add an internal captured-call type, for example toolproxy.CapturedCall or copilotgw.ToolCall, with Kind, SDKName, ResponseName, Namespace, CallID, ArgumentsJSON, Input, Execution, and raw metadata.
  • Keep Chat Completions adapted to []openai.ChatToolCall; make Responses serialization use the generic call type.
  • Add an internal continuation-output type with Kind, CallID, Output, optional Name, Status, Execution, and returned Tools.
  • Thread these types through parseResponsesInput, copilotgw.ResponseRequest, pending-batch matching, batch completion, fallback transcript construction, and response serialization.
  • When the SDK invokes a flattened namespace tool, store the namespace and child name so the Responses item can be rehydrated.
  • When the SDK invokes a custom tool, unwrap {"input":"..."} into a raw input string; if malformed, return a model-visible tool error.
  • When the SDK invokes tool_search, preserve arguments as structured JSON rather than string-only function arguments.
Phase 4

Rehydrate Responses output item types

Update the gateway and stream encoder to emit the item type expected by Codex CLI, with stable call_id values for every call kind.

  • Add fields or custom marshaling to openai.ResponseOutputItem for Namespace, Input, Execution, and raw/structured arguments for tool_search_call.
  • Ensure function_call.arguments remains a JSON string while tool_search_call.arguments serializes as an object.
  • Emit direct function tools as {"type":"function_call","call_id":"call_...","name":"exec_command","arguments":"..."}.
  • Emit namespace children as {"type":"function_call","call_id":"call_...","namespace":"mcp__grep_app","name":"searchGitHub","arguments":"..."}.
  • Emit custom tools as {"type":"custom_tool_call","call_id":"call_...","name":"apply_patch","input":"..."}.
  • Emit discovery as {"type":"tool_search_call","call_id":"call_...","execution":"client","arguments":{...}} with no name or namespace.
  • Persist the same rehydrated output items in stored Responses records so restart/TTL fallback sees canonical Responses shapes, not SDK aliases.
Phase 5

Support continuations for new output kinds

Codex CLI will answer custom and discovery calls with different input item types than normal function calls.

  • Extend ResponseInputItem and/or retain per-item raw JSON for namespace, status, execution, tools, and optional name on custom_tool_call_output.
  • Extend parseResponsesInput to recognize custom_tool_call_output and tool_search_output in addition to function_call_output.
  • Reuse outputRawToString for custom_tool_call_output.output, preserving support for text parts and structured content; tolerate and validate optional name when present.
  • For tool_search_output, require the matching call_id, validate execution: "client" for client-owned calls, parse status, and parse tools as Codex loadable specs.
  • Limit tool_search_output.tools to loadable function and namespace specs. Reject or ignore-with-diagnostic hosted, custom, nested namespace, and tool_search returned specs according to the validation matrix.
  • Validate that the continuation item type matches the pending call kind for active matched IDs only, preserving existing Codex stateless-history behavior where stale old outputs can appear alongside one live batch.
  • Update fallback transcript reconstruction to include custom_tool_call_output, tool_search_call, tool_search_output, and namespaced function_call history when a live pending batch is unavailable.
Phase 6

Define tool_search as a turn-breaking continuation

tool_search_output.tools cannot make newly returned tools callable inside an already configured SDK turn. The implementation must make the boundary explicit rather than pretending SDK tools can mutate mid-turn.

  • When a live pending tool_search_call receives tool_search_output, do not simply unblock the same parked SDK tool handler and expect new tools to appear.
  • Record the search result in the transcript/persisted response, merge returned loadable specs into the normalized catalog, and start/resume a new SDK model turn configured with the merged catalog for the model’s post-search continuation.
  • If SDK/session constraints prevent this new-turn strategy in the first implementation slice, explicitly narrow behavior to: accept and persist tool_search_output, then make returned tools available only when the client resubmits them in tools on a subsequent fresh request.
  • Add tests that prove returned tools are not claimed callable until the new tool catalog has actually been installed in the SDK session configuration.
Phase 7

Preserve safety invariants

Do not accidentally turn this into a proxy-hosted tool runtime.

  • Continue creating the Copilot client with ModeEmpty.
  • Keep AvailableTools restricted to request-scoped synthetic custom tools or the impossible sentinel.
  • Keep permission approval limited to exact request-scoped SDK names.
  • Honor tool_choice: "none" by withholding all synthetic tools, including custom, namespace, and tool_search.
  • Do not configure proxy-hosted MCP servers, skills, shell, repository operations, memory, file access, hosted web search, or hosted image generation.
  • Add regression tests proving SDK built-ins remain unavailable when namespace MCP tools are supplied by Codex CLI.
Phase 8

Update Responses WebSocket and warm-session paths

The feature targets Responses compatibility, so the WebSocket transport and generate:false warm sessions must either support the same normalized catalog or be explicitly documented as out of scope.

  • Replace warm-session storage/comparison of openai.SupportedTools(req.Tools) with the normalized client-tool catalog.
  • Ensure WebSocket response.create accepts the same extended tool specs, emits the same rehydrated output items, and handles extended-tool continuations with the same validation rules as HTTP.
  • Add warm-session reuse and mismatch tests for normalized catalogs, including custom, namespace, dotted function, and tool_search tools.
  • If WebSocket cannot be completed in this slice, add a documented feature gate and explicit tests proving unsupported extended-tool WebSocket requests fail clearly rather than silently filtering tools.
Phase 9

Add Codex-facing model metadata only if request capture proves it is needed

Some Codex tools are gated before the request by Codex model metadata, notably apply_patch_tool_type and search-tool support. The first implementation should not guess; it should capture whether Codex actually sends the missing tool specs and which model endpoint shape it uses.

  • If Codex is already sending custom, namespace, and tool_search, do not alter model responses for this feature.
  • If Codex omits apply_patch or tool_search because metadata is absent, add an opt-in Codex metadata compatibility mode rather than changing the OpenAI model list shape globally.
  • Candidate metadata fields: apply_patch_tool_type: "freeform", supports_search_tool: true, supports_parallel_tool_calls: true, and relevant context/truncation hints.
  • Prefer adding these under data[].metadata only if Codex is actually consuming the OpenAI model-list envelope; otherwise implement the exact Codex model endpoint shape Codex CLI requests.
  • Document and test any new config knob such as COPILOT_CODEX_MODELS_COMPAT=1, and prove default /v1/models output is unchanged.

Proposed data model

type ResponsesToolKind string

const (
    ToolKindFunction   ResponsesToolKind = "function"
    ToolKindCustom     ResponsesToolKind = "custom"
    ToolKindNamespace  ResponsesToolKind = "namespace"
    ToolKindToolSearch ResponsesToolKind = "tool_search"
)

type NormalizedTool struct {
    Kind         ResponsesToolKind
    Name         string          // public function/custom/tool_search name or namespace name
    Namespace    string          // set for namespace children
    Description  string
    Parameters   json.RawMessage // JSON schema when applicable
    Format       json.RawMessage // custom/freeform grammar metadata
    Execution    string          // tool_search execution, usually "client"
    Strict       *bool
    DeferLoading *bool
    Children     []NormalizedTool // namespace children
    Raw          json.RawMessage
}

type ClientTool struct {
    SDKName      string          // safe SDK-facing alias
    ResponseKind ResponsesToolKind
    ResponseName string          // canonical Responses-facing name
    Namespace    string
    Description  string
    Parameters   map[string]any
    Strict       *bool
    DeferLoading *bool
    Raw          json.RawMessage
}

type CapturedToolCall struct {
    Kind          ResponsesToolKind
    SDKName       string
    ResponseName  string
    Namespace     string
    CallID        string
    ArgumentsJSON json.RawMessage // function/namespace/tool_search args
    Input         string          // custom/freeform raw input
    Execution     string          // tool_search execution
}

type ResponseToolOutput struct {
    Kind      ResponsesToolKind
    CallID    string
    Name      string          // optional custom_tool_call_output name
    Output    string
    Status    string          // tool_search_output status
    Execution string          // tool_search_output execution
    Tools     json.RawMessage // tool_search_output loadable specs
}

The exact names can change during implementation; the important property is that the SDK-facing name is an implementation detail, while the Responses-facing identity is preserved losslessly for the client. Chat Completions can continue adapting captured calls to []openai.ChatToolCall; Responses should use the generic captured-call and output types directly.

Streaming and continuations

Existing function-call streaming emits response.output_item.added, argument deltas, argument done, and output item done. For new call types, the minimum compatible stream should still announce each item before completing it: emit response.output_item.added, then response.output_item.done, then response.completed. Granular deltas are optional for the first implementation.

Call kind Non-stream response output item Streaming minimum Continuation input
Function function_call with call_id Existing function call lifecycle function_call_output
Namespace child function_call with call_id and namespace response.output_item.added + response.output_item.done function_call_output
Custom/freeform custom_tool_call with call_id response.output_item.added + response.output_item.done custom_tool_call_output
Tool search tool_search_call with call_id and execution response.output_item.added + response.output_item.done tool_search_output
Future enhancement: after basic compatibility lands, add richer delta events if Codex or other clients require them. Use the exact Codex custom input event name response.custom_tool_call_input.delta for custom/freeform streaming; do not confuse it with response.function_call_arguments.*.

Codex model metadata considerations

The proxy’s OpenAI-compatible /v1/models endpoint is intentionally shaped like OpenAI’s model list. Codex, however, has model metadata fields that influence tool planning. This feature should not globally change the model-list contract without evidence.

  1. Measure first: capture an actual Codex run and confirm both whether the missing tools are present in tools and which model endpoint shape Codex requested.
  2. If present: the tool-filtering implementation is sufficient; leave model metadata alone.
  3. If absent: add an opt-in compatibility mode, for example COPILOT_CODEX_MODELS_COMPAT=1, that augments only the model response shape Codex is actually consuming.
  4. Do not overclaim: only advertise capabilities that the proxy can round-trip safely. For example, advertise apply_patch_tool_type only after custom_tool_call support is implemented, and advertise search-tool support only after the tool_search_output lifecycle is implemented or explicitly scoped.
  5. Document the knob: if a compatibility mode is added, update the README config table and add tests proving default model output is unchanged.

The implementation should make validation behavior explicit before runtime changes land. Supported client-owned Responses tool types are accepted in both modes; unsupported hosted/proxy-executed semantics fail closed when accepting them would mislead the client.

Test plan

Unit tests

  • Unmarshal and validate function, custom, namespace, and top-level tool_search specs.
  • Update existing tests that intentionally assert old function-only behavior, such as permissive Codex defaults and unsupported-only sentinel tests.
  • Reject malformed namespaces, duplicate flattened SDK names, nested namespaces, malicious returned loadable specs, and unsupported hosted tools according to the validation matrix.
  • Verify SDK-safe aliasing for unsafe function names such as multi_tool_use.parallel, custom tools, namespace children, and tool_search.
  • Verify custom apply_patch schema synthesis and input unwrapping.
  • Verify strict, defer_loading, raw format, and raw returned loadable specs are preserved.

HTTP/gateway tests

  • Responses request with mixed tool kinds passes all supported normalized tools to the gateway.
  • Non-stream response serializes custom_tool_call, namespaced function_call, and tool_search_call with stable call_id values.
  • SSE stream emits output_item.added before output_item.done for each new call kind, with stable output indexes when reasoning/message items precede tools.
  • Continuations accept custom_tool_call_output and tool_search_output, including optional custom output name and tool-search status/execution/tools.
  • Wrong output kind for a pending call kind, missing call_id, duplicate live call_id, model mismatch, and previous-response mismatch fail clearly.
  • Cold fallback/restart/TTL paths reconstruct transcripts for custom calls, namespaced calls, tool-search calls, and their outputs.

Safety and transport tests

  • No request-scoped tools results in the impossible sentinel.
  • tool_choice: "none" with function, custom, namespace, and tool_search tools withholds all synthetic SDK tools.
  • Supplying MCP namespace tools does not configure proxy-hosted MCP servers.
  • Permission handler approves only exact synthetic SDK names.
  • SDK built-ins remain disabled with ModeEmpty.
  • Responses WebSocket response.create and generate:false warm sessions store, compare, and reuse normalized extended tool catalogs—or fail clearly behind an explicit scope gate.

End-to-end and deterministic integration tests

  • Use a fake SDK/provider capture to assert exact SDK Tools and AvailableTools names, descriptions, schemas, and request-scoped filters.
  • Run Codex CLI against the proxy and ask it to list tools as a smoke test, while treating model output as nondeterministic.
  • Assert the smoke-test list includes apply_patch, at least one mcp__* namespace child tool, and top-level tool_search.
  • Perform a small apply-patch round trip in a disposable workspace.
  • Perform one MCP namespace child round trip using a fake local MCP server controlled by the test.
  • Perform a tool_search round trip and prove returned tools become callable only after the implementation’s explicit new-catalog boundary.

Rollout and risks

Risk Impact Mitigation
SDK name encoding changes model behavior The model may list or call encoded names instead of Codex’s canonical names. Keep names as close as safely possible; include canonical namespace/name in descriptions; rehydrate output names regardless of SDK name.
Freeform tools are represented as JSON schema to the SDK The model may wrap apply_patch incorrectly. Use a single input field with strong description. Add prompt/schema tests and fallback parsing for common shapes.
Tool-search requires dynamic tool availability Loaded tools may not be callable in the same SDK turn. Implement the explicit new-turn/new-catalog boundary or narrow the scope until it is available; never claim returned tools are callable before SDK configuration proves it.
Accidental proxy-side tool execution Breaks security expectations and project scope. Handlers continue to park and return client outputs only; add explicit safety regression tests.
OpenAI compatibility regressions Non-Codex clients may see changed behavior. Keep Chat Completions function-only; document the Responses extended-tool behavior, validation matrix, and hosted-tool fail-closed behavior.
WebSocket/warm-session drift HTTP works but Codex WebSocket or generate:false silently filters tools or disconnects warmed sessions. Update warm-session catalog comparison and add WebSocket coverage, or explicitly gate unsupported transport paths.

Recommended rollout is default-on for supported client-owned Responses tools once tests pass. Strict mode should accept the newly supported client-owned types and still reject malformed, hosted, proxy-executed, or unknown unsafe semantics. README updates are part of the rollout, including the validation matrix, tool-search boundary, WebSocket scope, and any Codex model metadata compatibility knob.

Acceptance criteria

  1. A mixed Codex Responses request containing function, custom, namespace, and top-level tool_search tools no longer filters the latter three out.
  2. Unsafe Responses function names such as multi_tool_use.parallel are SDK-aliased safely and rehydrated to their original public names.
  3. Codex CLI can see and name the missing tools when asked to list available tools through this proxy: apply_patch, at least one mcp__* namespace child tool, and tool_search.
  4. Every emitted call item has a stable unique call_id that matches pending batches, stored responses, streamed items, and continuation input items.
  5. apply_patch produces a custom_tool_call with raw input and accepts a custom_tool_call_output continuation.
  6. A namespaced MCP child tool produces a function_call with namespace and accepts a normal function_call_output.
  7. tool_search produces a tool_search_call, accepts tool_search_output, and either installs returned loadable tools through an explicit new-catalog boundary or clearly scopes returned tools to a later fresh request.
  8. tool_search_output.tools accepts only loadable function and namespace specs and applies the same aliasing, collision, and safety validation as initial request tools.
  9. Streaming emits response.output_item.added before response.output_item.done for new call kinds, with stable sequence numbers and output indexes.
  10. Responses WebSocket and warm-session paths either support normalized extended tools with tests or explicitly fail/gate unsupported extended-tool requests with clear errors.
  11. tool_choice: "none" with extended tools exposes no synthetic SDK tools.
  12. No proxy-hosted shell, file, MCP, memory, skill, repository, hosted web-search, or hosted image-generation tools become available unless they were supplied by the client as request-scoped client-owned tool specs.
  13. Strict and permissive behavior matches the validation matrix and is documented in README.
  14. If Codex model metadata compatibility is implemented, it is opt-in, documented, tested, and does not change default /v1/models output.
  15. Existing Chat Completions function-tool behavior remains unchanged.
