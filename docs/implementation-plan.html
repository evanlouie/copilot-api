Implementation Plan · copilot-api

OpenAI-compatible API gateway for GitHub Copilot models.

A phased plan to build a Go service that exposes OpenAI Chat Completions and Responses API compatible endpoints backed by the GitHub Copilot SDK, while behaving like a raw/vanilla inference provider: no Copilot default persona, no Copilot SDK tools by default, client-owned tool calls, streaming support, model discovery, and a clear path toward richer compatibility.

01Goals and boundaries

The project should let a single user point multiple concurrent OpenAI-compatible clients at a local proxy base URL and use models available through their GitHub Copilot subscription. The first release should prioritize a dependable compatibility core over claiming every OpenAI feature immediately.

Primary goals

  • Expose GET /v1/models, POST /v1/chat/completions, and POST /v1/responses.
  • Support both buffered JSON responses and OpenAI-style server-sent event streams.
  • Map Copilot SDK model metadata, assistant text events, streaming deltas, usage events, and errors into OpenAI-shaped responses.
  • Never use the Copilot SDK default system prompt, coding-agent persona, or SDK-managed behavioral instructions. The effective model instructions must be only the caller-provided OpenAI system/developer/instructions content plus the minimum neutral adapter framing required for protocol translation.
  • Disable all Copilot SDK tools by default. The runtime must not read files, write files, run shell commands, call MCP servers, access memory, or invoke any SDK-provided tool unless a future explicit opt-in feature enables a narrowly documented capability.
  • Support client-defined OpenAI tools by proxying their schemas into temporary SDK custom tools and yielding tool invocations back to the client for execution; the API broker never runs client business logic itself.
  • Work with standard OpenAI SDKs by changing base_url and, when proxy auth is enabled, the API key.

Non-goals for the first milestone

  • Perfect parity with every OpenAI request field, beta endpoint, fine-tuning endpoint, batch endpoint, audio endpoint, or embeddings endpoint.
  • Bypassing Copilot subscription rules, quotas, model policies, or authentication requirements.
  • Using Copilot as a coding agent, preserving the Copilot CLI persona, or exposing Copilot SDK tools as part of the default API behavior.
  • Executing arbitrary OpenAI tool calls on the server. The server brokers tool requests and results; clients own execution.
  • Hiding differences in model behavior caused by Copilot SDK agent/session semantics.
Compatibility posture: the Copilot SDK is an upstream model transport, not the product persona. Implement a small, well-tested OpenAI-compatible surface first; replace the SDK default prompt, keep SDK tools unavailable by default, accept unsupported options only when they can be safely ignored, and return OpenAI-shaped invalid_request_error responses when ignoring would surprise clients.

02Resolved design decisions

The implementation plan has been refined through design review and two focused spikes: spikes/synthetic-session-hydration.md and spikes/tool-choice-enforcement.md. These decisions are part of the product contract unless a later implementation spike explicitly supersedes them.

Runtime and state model

  • The service is a single-user concurrent proxy: many clients and sessions may be active, but there is one Copilot account/auth context.
  • There is no ambient or global conversation. Continuity requires explicit protocol state such as previous_response_id or proxy-issued pending tool-call IDs.
  • Chat Completions is stateless and transcript-driven. Ordinary Chat requests use synthetic Copilot session hydration as the default history path.
  • Responses is the higher-fidelity stateful API and reuses the stored SDK session via previous_response_id.
  • Retained session state is for observability/debugging, not implicit reuse.

Prompt and SDK behavior

  • Always pass SystemMessageConfig{Mode: "replace"}. Never pass nil, append, or customize mode.
  • Leading Chat system/developer messages and Responses instructions are combined into the replacement prompt. Mid-conversation system/developer messages are rejected.
  • If no caller instructions exist, try empty replacement, then single-space replacement, then exact minimal fallback: You are a chat completion model.
  • Use the SDK-bundled/matched Copilot CLI by default. COPILOT_CLI_PATH is an explicit advanced override.
  • SDK/CLI upgrades are blocked unless hydration, tool, prompt-isolation, and provider-shape tests pass.

Tools

  • Copilot SDK built-in tools, proxy-hosted MCP, config discovery, skills, memory, shell, file, and repository tools are disabled by default.
  • Clients may define OpenAI function tools. The proxy registers opaque request-scoped SDK aliases and yields tool calls back to clients; it never executes client business logic.
  • Client-hosted MCP is supported when the client exposes MCP tools as OpenAI function tools. Proxy-hosted MCP is not part of MVP.
  • MVP supports tool_choice omitted/auto and none. Forced function and required are rejected because the SDK/CLI does not enforce them.
  • Tool outputs are string-first. JSON object/array outputs are accepted as a convenience and serialized to JSON text; binary/multimodal outputs are rejected in MVP.

Operations and storage

  • model is required on every generation request. Unknown models are rejected after one forced model-cache refresh.
  • Proxy auth is opt-in: if COPILOT_API_KEY is set, require bearer auth; if unset, start without auth and log a warning. Default bind remains loopback.
  • No proxy-level concurrency limits, generation timeouts, or payload limits by default. Pending tool-call TTL remains a configurable liveness guard.
  • Use XDG directories for data/state/cache/config. Retain session state verbatim forever by default.
  • Include copilot-api purge [--dry-run] [--yes] in MVP. Purge refuses while a server lock is active; the same lock prevents two servers using the same store.
Thin proxy principle: where behavior is not required for OpenAI compatibility or safety, prefer passing through to the SDK/service over adding project-specific throttles, schedulers, or policy layers.

03Proposed architecture

Keep the Copilot SDK behind narrow internal interfaces, and isolate the risky parts: synthetic session hydration, client-tool proxying, and SDK session storage. The HTTP layer should remain OpenAI-shaped while the gateway owns SDK lifecycle and version coupling.

HTTP/API layer

  • Routes, request parsing, optional bearer auth, request IDs, and OpenAI-compatible error envelopes for /v1/*.
  • SSE response writing with complete buffered tool-call objects in MVP.
  • Simple /healthz and /readyz operational JSON.

Compatibility layer

  • OpenAI Chat Completions and Responses structs, validation, IDs, usage, and error mapping.
  • Instruction folding into replacement system prompts.
  • Tool schema aliasing, pending tool-call batches, and strict tool-output validation.

Copilot integration layer

  • Embedded/matched CLI startup, ping, readiness, model cache, and SDK client lifecycle.
  • Synthetic Chat hydration through a project-owned SessionFsProvider.
  • Responses SDK session reuse, parked tool handlers, and event collection.

05Client-defined tool-call proxy plan

Tool calls follow the raw OpenAI provider model: the client declares tools, the model decides whether to call them, the API returns the call to the client, and the client executes the tool and sends the result back. The server is only a schema adapter and synchronization broker.

Client-owned execution

Only tools supplied in the OpenAI request are exposed. The proxy registers opaque SDK aliases and yields calls; clients perform side effects, including MCP if they choose to back a function tool with MCP.

No SDK built-ins

File, shell, MCP, memory, skill, repository, and other Copilot SDK tools stay unavailable. Unknown SDK tool execution is an invariant violation and aborts the turn.

Parked continuations

When tools are called, Chat and Responses park live SDK proxy handlers and wait for exact client outputs, bounded by COPILOT_TOOL_CALL_TTL.

Inbound schema conversion

  1. Accept only OpenAI function tools in MVP. Reject hosted tools, proxy-hosted MCP, file search, web search, computer/browser tools, and non-function tool types.
  2. Validate function name, description, JSON Schema parameters, optional strict, and request shape. Do not impose proxy-specific tool-count or payload-size limits by default.
  3. Register every client tool under an opaque internal SDK alias and map it back to the public OpenAI name in all client-visible responses.
  4. Set AvailableTools to exactly the request-scoped aliases, or to an impossible sentinel for tool_choice: "none". Continue denying all SDK-provided or unknown permission requests.
  5. Support tool_choice omitted/auto and none. Reject forced function and required because the SDK/CLI does not expose true enforcement.

Proxy handler contract

// Created per request from OpenAI function tool schemas.
func proxyToolHandler(inv copilot.ToolInvocation) (copilot.ToolResult, error) {
    call := PendingToolCall{
        OpenAIToolCallID: mapSDKCallID(inv.ToolCallID),
        SDKToolCallID:    inv.ToolCallID,
        PublicName:       mapInternalNameToPublic(inv.ToolName),
        ArgumentsJSON:    preserveArgumentsJSON(inv.Arguments),
    }

    publishCompleteToolCall(call)       // yields OpenAI tool_calls/function_call output
    result := waitForClientToolOutput() // parked until all batch outputs arrive or TTL expires
    return toolResultForLLM(result), nil // string passthrough or JSON text
}

Chat Completions behavior

StepServer behaviorClient behavior
Request with tools Hydrate prior Chat history, resume synthetic session with replacement prompt, register proxy aliases, and send the final user turn. Send standard OpenAI tools with tool_choice omitted/auto or none.
Model calls tools Capture one or more SDK proxy invocations, return complete OpenAI message.tool_calls, preserve any pre-tool text, set finish_reason to tool_calls, and keep the SDK handlers parked. Execute the named tools locally using returned arguments.
Continuation Recognize returned tool_call_ids as explicit continuation handles, validate exactly one output per pending call, unblock the parked handlers, and let the same SDK turn continue. Append one role: "tool" message per returned call and call /v1/chat/completions again before TTL expiry.
Streaming Stream text deltas, buffer tool invocations, emit complete tool-call objects near the end, terminal chunk with finish_reason: "tool_calls", then data: [DONE]. Accumulate text/tool calls, execute tools after the stream ends, then submit outputs.

Responses API behavior

  1. Create or resume the stored SDK session for the response chain; reuse that session for previous_response_id.
  2. When a proxy tool is invoked, return one or more function_call output items with stable call_id, public name, and arguments.
  3. The client executes tools and submits function_call_output items with previous_response_id.
  4. The server validates ownership, unknown/missing/duplicate IDs, and one output per pending call, then unblocks the matching SDK proxy handlers.
  5. If pending state expires, mark retained metadata as expired, abort/disconnect the live SDK session, retain files for debugging, and return an OpenAI-shaped expired-call error on future continuation.

Tool output and error semantics

Output shape

  • String outputs pass through exactly to ToolResult.TextResultForLLM.
  • JSON object/array outputs are accepted as a convenience and serialized to JSON text, preserving raw JSON bytes where possible.
  • Binary, image, audio, file-reference, and partial/streaming tool outputs are rejected in MVP.

Errors and batches

  • Client tool failures are normal tool output content, not HTTP/proxy failures.
  • Parallel batches are supported internally; all pending outputs for a batch are required before continuation.
  • Unknown SDK tool calls, built-in tool requests, expired pending calls, and aliasing failures are invariant/upstream errors and are logged with request IDs.
  • Pending TTL is configurable; it is a live-handler liveness guard, not a debug-data retention policy.

07Request and event flows

Chat Completions flow

  1. If COPILOT_API_KEY is set, authenticate Authorization: Bearer ...; if unset, run unauthenticated and log a warning.
  2. Validate model, message ordering, feature support, client tool schemas, tool_choice, parallel_tool_calls, and any tool-output continuation.
  3. For ordinary Chat turns, split leading system/developer messages into replacement prompt, hydrate prior messages into versioned SDK events, resume the synthetic session, and send only the final user message.
  4. For recognized parked tool continuations, validate every returned tool_call_id against pending state and unblock the live SDK proxy handlers instead of sending a new prompt.
  5. Use disabled SDK built-ins, request-scoped proxy tool aliases, no config discovery/MCP/skills, and a denying permission handler with exact proxy-tool exceptions.
  6. Return OpenAI JSON or SSE frames. If proxy tools are invoked, yield complete tool_calls, preserve pre-tool text, and finish with tool_calls.
  7. Retain synthetic session state and metadata.json forever under XDG data for debugging; never reuse it implicitly for later Chat requests.

Responses API continuity flow

  1. If previous_response_id is absent, create a new stored SDK session and response record.
  2. If present, load the stored mapping and resume/reuse the same SDK session with the same safety settings.
  3. For function_call_output items, validate ownership, unknown/missing/duplicate IDs, and one output per pending call, then unblock matching proxy handlers.
  4. For normal new input, send only the new input to the resumed SDK session.
  5. Persist response metadata, SDK session ID, model, status, output items, pending call IDs, usage, and API-visible store state. Debug files remain retained forever even when store:false disables later retrieval.

09Security and operations

Security defaults

  • Proxy auth is enabled only when COPILOT_API_KEY is set; otherwise the server starts unauthenticated and logs a clear warning. Default bind is 127.0.0.1:8080.
  • Always replace the Copilot SDK system prompt; never use SDK default, append, customize, or nil system prompt.
  • Deny Copilot permission requests by default; allow only exact request-scoped proxy tools whose handlers perform no side effects.
  • Disable SDK tools, proxy-hosted MCP, config discovery, skills, repository instructions, shell/file access, and memory by default.
  • Use logged-in GitHub/Copilot auth by default; GITHUB_TOKEN is an optional process-wide override. No per-request GitHub auth in MVP.

State and lifecycle

  • Use XDG data/state/cache/config directories with explicit env overrides. Create sensitive dirs/files with 0700/0600 best effort.
  • Retain session state verbatim forever by default for debugging. Store proxy metadata.json next to every retained Chat/Responses session.
  • Use a server lock to prevent two processes sharing the same store and to make copilot-api purge safe.
  • Do not implement scheduled cleanup. Manual purge is the user-controlled cleanup mechanism.
  • Pending tool-call TTL remains configurable and aborts live parked handlers while retaining debug files.

Thin pass-through operations

  • No proxy-level concurrency limits, generation timeouts, or payload limits by default.
  • Startup/readiness and model-refresh operations may have practical timeouts because startup cannot hang forever.
  • Use embedded/matched CLI by default; fail startup/readiness on SDK/CLI incompatibility or model-list failure.
  • Logs redact prompts, responses, tool arguments, tool outputs, and tokens by default. COPILOT_LOG_CONTENT=true enables content logging with a loud warning.
  • Every HTTP response includes X-Request-ID; client-supplied request IDs are metadata only.

10Risks and mitigations

RiskImpactMitigation
Synthetic hydration depends on internal SDK event semantics SDK/CLI changes could silently break Chat history fidelity. Pin SDK/CLI, isolate internal/hydration, validate generated events, retain transcript fallback as an explicit escape hatch, and gate upgrades on live hydration tests.
Copilot default prompt or tools leak into requests The API would stop behaving like a vanilla inference provider. Always replacement-mode system prompts; empty/space/minimal fallback tests; layered built-in tool disablement; abort on unknown SDK tool execution.
OpenAI tool-choice semantics cannot be fully honored Clients using forced function or required tool use could get direct text despite expecting a tool call. Support only auto/none in MVP and reject forced/required with OpenAI-shaped errors; document spike evidence.
Parked tool-call continuations leak live resources Blocked SDK handlers could wait forever if clients never return outputs. Use configurable pending TTL, abort/disconnect expired live sessions, retain expired state for debugging, and reject late continuations clearly.
Forever retention stores sensitive data Prompts, responses, and tool outputs remain on disk until manual purge. Use XDG user dirs with restrictive permissions, document sensitivity and purge command, redact logs by default, and never expose Chat retrieval endpoints.
Multiple server processes corrupt the store Concurrent writers could damage retained session state or pending records. Acquire a server lock in XDG state dir; purge refuses while lock is held; users can run multiple instances only with distinct dirs/ports.
