WebSocket mode for long-running Responses workflows

A phased plan to add OpenAI-compatible WebSocket transport to /v1/responses, reusing the existing Responses validation, streaming event model, Copilot SDK gateway, tool-call continuation machinery, and prompt isolation guarantees.

Summary

OpenAI's Responses WebSocket mode keeps a persistent connection to /v1/responses. Clients send response.create events over the socket, and the server returns the same Responses streaming event model already used by SSE. The biggest behavioral difference is continuation: the active socket should retain the most recent response state in memory so store:false chains can continue without writing API-visible response state to disk.

Recommended strategy: add a WebSocket transport layer in internal/httpapi that converts socket messages into the existing copilotgw.ResponseRequest path, while extracting shared stream-event serialization from the current SSE handler. Keep the gateway as the single source of generation behavior.

Reuse

Validation, input parsing, reasoning-effort resolution, tool registration, session runners, and OpenAI response event schemas.

Add

WebSocket upgrade handling, client event parsing, per-connection sequencing, socket-local response cache, and JSON event writer.

Defer

Standalone /v1/responses/compact unless separately prioritized; document it as unsupported in the compatibility matrix.

Protocol contract

The first compatibility target is the WebSocket mode documented at OpenAI API: WebSocket Mode. The project should support the documented core loop before adding extra conveniences.

  "model": "gpt-5.5",
  "store": false,
  "previous_response_id": "resp_123",
  "input": [
    {
      "type": "function_call_output",
      "call_id": "call_123",
      "output": "tool result"
    }
  ],
  "tools": []

Architecture

Treat WebSocket mode as a transport adapter. The HTTP handler should be responsible for upgrade, event IO, connection lifetime, and socket-local state. The gateway should remain responsible for model validation, Copilot SDK sessions, response construction, tool batches, and persistence. 

flowchart TD
    Client[WebSocket client] -- response.create --> WS[httpapi responses WebSocket handler]
    WS --> Decode[Decode and validate client event]
    Decode --> Parse[Reuse Responses input parsing and reasoning resolution]
    Parse --> CacheCheck{previous_response_id in socket cache?}
    CacheCheck -- yes --> Fast[Use connection-local continuation metadata]
    CacheCheck -- no --> Store[Fallback to persisted response state when store=true]
    Fast --> GW[copilotgw.StreamResponse]
    Store --> GW
    GW --> Events[ResponseStreamEvent channel]
    Events --> Shape[Transport-neutral Responses event shaper]
    Shape -- JSON frames --> WS
    WS --> Client

Proposed file map

  • internal/httpapi/server.go — route GET /v1/responses to a WebSocket upgrade handler before the existing response-id subtree route.
  • internal/httpapi/responses_ws.go — WebSocket upgrade, read loop, sequential execution guard, connection deadline, close/error mapping.
  • internal/httpapi/responses_stream.go — transport-neutral conversion from copilotgw.ResponseStreamEvent to openai.ResponseStreamEvent values.
  • internal/openai/websocket.go — client event envelope structs, WebSocket error envelope helpers, and optional close-code constants.
  • internal/copilotgw/responses.go — minimal extension points for socket-local continuation records if transport-only adaptation is insufficient.
  • internal/sessionstore/store.go — no required schema change for MVP; add helpers only if persisted fallback needs clearer semantics.
  • internal/httpapi/server_test.go or responses_ws_test.go — protocol, event ordering, continuation, and error tests.
Dependency decision: use github.com/coder/websocket unless there is a strong reason to avoid a new dependency. It is small, actively maintained, context-aware, and avoids the older Gorilla package's maintenance history. Add it with go get, then run goimports -w after edits.

State model

OpenAI WebSocket mode's main value is connection-local continuation. The implementation should keep a small in-memory record for the latest response on each socket and use persisted state only as a fallback when allowed.

Case Expected behavior Implementation detail
No previous_response_id Start a new response chain. Create a fresh resp_ ID and SDK session exactly like HTTP streaming.
Previous ID equals socket cache Continue quickly, including with store:false. Use connection-local metadata to resume the SDK session or to build a synthetic continuation prompt if resume fails.
Previous ID not cached, store:true Fallback to persisted response metadata when available. Reuse current LoadResponseForContinuation / Stored checks for normal persisted chains.
Previous ID not cached, store:false Return previous_response_not_found. Do not rely on debug-retained disk files to emulate ZDR-compatible behavior.
Continuation fails with 4xx/5xx Evict the referenced cached previous response. Delete the socket cache entry before sending the terminal error event.
Design tension: the current gateway stores response records for debug retention even when store:false. WebSocket mode should not use that debug persistence as a continuation source for uncached store:false responses. Keep API-visible storage, debug retention, and socket-local cache as separate concepts.

Implementation phases

Establish shared streaming event shaping

Extract the logic in streamResponses that turns internal gateway events into OpenAI Responses stream events. The new helper should not know whether events are written as SSE lines or WebSocket JSON frames.

  • Create a transport interface such as responseEventSink with Write(event openai.ResponseStreamEvent) error and Done() error.
  • Move message item bookkeeping, output-text done events, function-call argument events, and failure events into the helper.
  • Keep current SSE output byte-for-byte equivalent where practical; update focused tests if event ordering becomes more explicitly normalized.

Add WebSocket upgrade and base event loop

Add GET /v1/responses support that accepts WebSocket upgrades and reads JSON messages from the client until close, deadline, or protocol error.

  • Add the WebSocket dependency with go get github.com/coder/websocket.
  • Implement responsesWebSocket(w, r) in internal/httpapi/responses_ws.go.
  • Use existing auth, logging, panic recovery, and request ID middleware.
  • Bound message size using COPILOT_MAX_REQUEST_BODY_BYTES or a smaller WebSocket-specific limit if added.
  • Reject unknown event types with a WebSocket JSON error event, not a raw HTTP error after upgrade.

Map response.create into existing request handling

Decode the WebSocket event into the existing openai.ResponsesRequest shape, then reuse validation, input parsing, tool filtering, and reasoning-effort resolution.

  • Add an envelope type with type, generate, and embedded Responses create fields.
  • Treat stream as implicit true and ignore any client-supplied value in permissive mode.
  • Preserve current strict compatibility behavior for unsupported semantic fields.
  • Log generation starts with a transport tag such as responses.websocket.

Enforce one in-flight response per connection

OpenAI's documented mode runs responses sequentially on a socket and does not multiplex. The implementation should make that invariant explicit.

  • MVP option: process the read loop synchronously and do not read the next response.create until the current turn completes.
  • If asynchronous reads are introduced later, guard with a mutex or state machine and reject overlapping work.
  • Cancel the active Copilot turn when the socket closes or the 60-minute connection limit is reached.

Implement socket-local continuation cache

Store the latest response record for the connection after a completed response or tool-call response. Use it before persisted fallback when previous_response_id is supplied.

  • Cache only what the gateway needs: response ID, SDK session ID, model, instructions, input text, output text/items, previous response ID, pending batch ID, retained path, and stored flag.
  • Keep only the most recent response for MVP to match the documented behavior. Consider a tiny LRU only if real clients need it.
  • For store:false, allow continuation only when the previous ID matches the connection cache.
  • On failed continuation, evict the referenced cached state before sending the error event.

Support tool-call-heavy continuation loops

Ensure function_call_output continuations over the socket unblock pending SDK handlers and emit subsequent response events on the same socket.

  • Reuse existing pending batch ownership validation: endpoint kind, model, previous response ID, and one output per pending call.
  • Make error messages match the documented previous_response_not_found and existing OpenAI-shaped invalid request envelopes.
  • Verify multiple tool calls in a batch and multiple sequential batches in one socket session.

Add lifecycle limits and observability

Add production controls before enabling the feature broadly.

  • Enforce a 60-minute maximum connection age and send websocket_connection_limit_reached before close when possible.
  • Emit structured logs for socket open, close, response start, response completion, response failure, cache hit, cache miss, and connection-limit closure.
  • Preserve content redaction rules; only include bodies when COPILOT_LOG_CONTENT=true.
  • Consider counters for active WebSocket connections and rejected overlapping requests if observability grows beyond logs.

Document compatibility and deferments

Update public documentation after tests pass.

  • Add WebSocket mode to the README endpoint and compatibility sections.
  • Document unsupported or deferred features: generate:false warmup if not implemented, standalone compaction, and any limits that differ from OpenAI.
  • Add examples using a minimal WebSocket client and a tool-call continuation loop.

Risks and mitigations

Risk Mitigation
Subtle drift between SSE and WebSocket event ordering Centralize event shaping and test both transports from the same synthetic gateway event stream.
store:false accidentally uses debug-retained disk state Require socket-cache hit for uncached store:false; test same-socket success and new-socket failure.
SDK session resume does not support desired fast path Reuse current resume behavior first; fallback to synthetic continuation prompt only where existing HTTP behavior already does.
Long-lived sockets hide stuck SDK turns Propagate context cancellation on socket close, enforce configured request timeout per turn, and enforce the 60-minute socket limit.
Tool-call batches leak if clients disconnect mid-turn Cancel active runner and remove connection-owned pending batches on socket close or terminal failure.
Dependency expands attack surface Choose a minimal WebSocket library, pin via go.mod, bound message sizes, and fuzz/error-test JSON input handling.

Acceptance criteria

Must pass

  • wss://host/v1/responses accepts authenticated WebSocket connections.
  • Client response.create events produce OpenAI-shaped Responses streaming JSON events.
  • Only one response runs at a time per connection.
  • Same-socket store:false continuation works with previous_response_id.
  • Uncached store:false continuation returns previous_response_not_found.
  • Tool-call continuation loops work over a single persistent socket.
  • go test ./... passes.

May defer

  • generate:false warmup, if no safe SDK warmup path is available.
  • Standalone /v1/responses/compact.
  • Multiplexing; OpenAI does not support it for this mode today.
  • Connection-local cache larger than one previous response.
  • Metrics beyond structured logs.
Definition of done: WebSocket mode is documented in the README compatibility matrix, covered by deterministic tests, and can run an agentic tool-call loop without requiring clients to reopen HTTP/SSE requests for each turn.
